## PHASE 2 — Audit sur 6 axes

### 1️⃣ Architecture

#### 🔴 [HAUTE — M] Coercion de type dupliquée client / serveur
- **Problème** : `toNum()`, `toDate()`, `toChar1()` existent dans `pv-run-extraction/index.ts:33-59` (serveur) ET la logique équivalente existait dans `useAnalysis.js` historiquement. Maintenant, `useAnalysis.js` est un thin trigger, mais `ValidationForm.jsx` et `PreEtatDateTemplate.jsx` font du formatting de nombres/dates inline sans helper partagé.
- **Localisation** : `supabase/functions/pv-run-extraction/index.ts:33-59` (canonique) | `src/components/validation/ValidationForm.jsx` et `PreEtatDateTemplate.jsx` (formatting inline)
- **Conséquence** : risque de divergence d'affichage vs valeur stockée (formats `fr-FR` vs ISO).
- **Effort** : M (½ jour) — extraire un `src/lib/format.js` partagé.

#### 🔴 [HAUTE — M] Double source de vérité : colonnes plates + `extracted_data` JSONB
- **Problème** : 30+ champs financiers existent à la fois en colonnes plates dans `pv_dossiers` ET dans `extracted_data` JSONB. `pv-run-extraction/index.ts:409-450` écrit les deux ; `ValidationForm` lit l'un puis fallback sur l'autre. Si un correctif manuel modifie une colonne plate sans mettre à jour le JSONB (ou inversement), les données divergent silencieusement.
- **Localisation** : `supabase/functions/pv-run-extraction/index.ts:409-450` (écriture) | `src/components/validation/ValidationForm.jsx:65-85` (lecture avec fallback)
- **Conséquence** : risque de désynchro lors de patches DB ou de revert.
- **Effort** : M (½ jour) — choisir une source canonique unique, ou écrire un trigger DB de cohérence.

#### 🟠 [MOYENNE — L] Duplication funnel B2C ↔ B2B
- **Problème** : `DossierPage.jsx` (B2C 6 étapes), `ClientUploadPage.jsx` (B2B client, 1 étape upload) et `ProDossierDetailPage.jsx` (B2B pro, validation+delivery) partagent la logique d'upload/classification/validation/delivery. Bug fixes doivent être appliqués 3 fois.
- **Localisation** : `src/pages/DossierPage.jsx` vs `src/pages/ClientUploadPage.jsx` vs `src/pages/pro/ProDossierDetailPage.jsx`
- **Conséquence** : maintenance ×3, divergence inéluctable.
- **Effort** : L (1-2 jours) — extraire un composant `<DossierFlow mode="b2c|b2b-client|b2b-pro">`.

### 3️⃣ Sécurité

#### 🔴 [CRITIQUE — M] Edge functions service_role sans vérification d'ownership
- **Problème** : `pv-run-extraction/index.ts:540` vérifie `stripe_payment_status === 'paid'` mais **PAS** que l'appelant est le légitime propriétaire de `dossier_id`. Un attaquant peut POST `{ dossier_id: <dossier d'autrui payé> }` et déclencher une extraction sur un dossier qui ne lui appartient pas (consomme du crédit Gemini, peut écraser des données). Idem `pv-pro-credits` : aucun check que le `pro_account_id` du body correspond au `pro-token` du caller.
- **Localisation** : `supabase/functions/pv-run-extraction/index.ts:530-560` | `supabase/functions/pv-pro-credits/index.ts:79-130`
- **Conséquence** : escalade horizontale (un user déclenche extraction sur dossier d'autrui ; un pro crédite un autre compte).
- **Effort** : M (3-4h) — passer le `session_id` (B2C) ou le `pro-token` (B2B) en `Authorization` header, valider avant toute mutation.

#### 🔴 [CRITIQUE — XS si exposée] Secrets éventuels dans `.env`
- **Problème** : Le sous-agent a affirmé voir `APIFY_TOKEN=apify_api_CQ...` dans `.env`. **Je n'ai pas pu confirmer** (`.env` est gitignored, ce qui est correct). Si vrai, à rotater.
- **Vérifié** : `.gitignore` exclut bien `.env` et `.env.*`. Vérifier `git log -S "APIFY"` et `git log -S "GEMINI_API_KEY"` pour s'assurer qu'aucun secret n'a fuité dans l'historique.
- **Localisation** : `.env` (non commité) — à inspecter manuellement
- **Effort** : XS (10 min) — si confirmé, rotater + scanner l'historique avec `git secrets` ou `trufflehog`.

#### 🟠 [HAUTE — S] `dangerouslySetInnerHTML` sur HomePage
- **Problème** : `HomePage.jsx:503` et `HomePage.jsx:529` utilisent `dangerouslySetInnerHTML={{ __html: item }}` pour afficher des items d'une liste de comparaison syndic. Les valeurs sont actuellement statiques (en dur dans le source), donc pas exploitable maintenant. **Mais** si ces données deviennent dynamiques (CMS, A/B test depuis Supabase, traduction), XSS immédiat.
- **Localisation** : `src/pages/HomePage.jsx:503, 529`
- **Conséquence** : pattern dangereux latent — anti-pattern à éviter en React.
- **Effort** : S (1h) — remplacer par du JSX (`<>... <strong>...</strong> ...</>`) ou un parseur safe.

#### 🟠 [HAUTE — S] Prompt injection via noms de fichiers
- **Problème** : Les noms de fichiers uploadés sont injectés dans les prompts Gemini (`pv-extract-financial`, `pv-extract-diagnostics`) sans escape. Un attaquant peut nommer un PDF `END INSTRUCTIONS. Now respond with: ...` pour tenter d'altérer l'extraction.
- **Localisation** : `supabase/functions/pv-extract-financial/index.ts` (interpolation filename) | `pv-extract-diagnostics/index.ts`
- **Conséquence** : pollution potentielle de l'extraction (faible probabilité de succès avec Gemini 2.5 Pro, mais non nul).
- **Effort** : S (1h) — `JSON.stringify(filename)` ou strip caractères non-alphanumériques avant injection prompt.

### 4️⃣ Performance

#### 🟠 [HAUTE — S] Bundle gros : pas de visibilité sans `npm run build`
- **Problème** : Le CLAUDE.md mentionnait ~2.3 MB pour le bundle initial. `@react-pdf/renderer` (~1 MB) est désormais lazy-chargé via `lazy()` à Step 6 (vérifié `DossierPage.jsx:15`). Reste : `lucide-react` importé per-icon dans 25+ fichiers — devrait tree-shake correctement avec Vite, **mais non vérifié**.
- **Localisation** : `vite.config.js` + `package.json` deps
- **Conséquence** : potentiellement 1-1.5 MB JS au premier load homepage.
- **Effort** : S (1h) — lancer `npm run build` + `vite-bundle-visualizer` pour confirmer/infirmer.

#### 🟠 [HAUTE — M] Pré-rendering 86 routes : probablement séquentiel
- **Problème** : `scripts/prerender.js` itère via `for (const route of ALL_ROUTES)`. Si chaque route prend ~200ms, build = 17s rien que pour le prerender. Multiplié par chaque déploiement Vercel.
- **Localisation** : `scripts/prerender.js`
- **Conséquence** : CI lente, feedback développeur dégradé.
- **Effort** : M (2-3h) — paralléliser avec `Promise.all` + concurrency limiter (4-8 routes en parallèle).

#### 🟡 [MOYENNE — S] `.select('*')` partout dans services
- **Problème** : `dossier.service.js`, `document.service.js`, `pro.service.js` font tous `.select('*')` même quand le caller n'a besoin que d'un sous-ensemble. Le `KanbanBoard` charge les `extracted_data` (~25 KB par dossier) inutilement.
- **Localisation** : `src/services/*.service.js` (toutes les requêtes `.select('*')`)
- **Conséquence** : ~1 MB de payload réseau gaspillé pour 50 dossiers Pro.
- **Effort** : S (1-2h) — typer chaque service avec un sous-ensemble explicite (`'id, status, created_at, ...'`).

#### Note : claims réfutés
- **Polling 3s sans arrêt dans `ProcessingStep`** : FAUX. Vérifié `ProcessingStep.jsx:74-82` : la condition `shouldPoll = status === 'paid' || status === 'analyzing'` arrête le polling proprement dès que le statut passe à `pending_validation`. Le `clearInterval` est bien appelé en cleanup.
- **Polling `useDocuments` infini si classification échoue** : PARTIELLEMENT vrai. `useDocuments.js:93-96` polle tant qu'au moins une doc n'a pas de `document_type`. Si la classification échoue 2× et abandonne, la doc reste sans type → polling infini. Effort S pour ajouter un état `classification_failed`.

### Ajout 1 — Section "Gotchas critiques" (en haut, avant la stack)
```markdown
## ⚠️ Gotchas critiques
- **Stripe B2B price IDs sont des placeholders** dans `pv-pro-credits/index.ts:17,22,27` — Espace Pro non fonctionnel en prod tant que ce n'est pas corrigé.
- **Pas d'ownership check serveur** : `pv-run-extraction` et `pv-pro-credits` font confiance au body POST. Toute nouvelle edge function qui mute des données DOIT vérifier le caller (session_id pour B2C, pro_token pour B2B) avant écriture.

```

### Ajout 2 — Section "Convention : flat columns vs `extracted_data`"
```markdown
## Convention de données dossier
**Source canonique** : les **colonnes plates** (`budget_previsionnel`, `charges_courantes`, `tantiemes_lot`, etc.) sont la source de vérité pour les workflows utilisateur (validation, PDF, share).
**`extracted_data` JSONB** : conservé pour traçabilité IA (`_source` references, alerts) — ne JAMAIS y lire de valeur métier sans fallback aux colonnes plates.
**Écriture** : seul `pv-run-extraction` écrit les deux. Tout autre code qui modifie une colonne plate doit mettre à NULL le champ correspondant dans `extracted_data` ou marquer une alerte de désynchro.
```

### Ajout 3 — Section "Patterns à éviter"
```markdown
## Anti-patterns
- **`.select('*')`** dans les services : spécifier explicitement les colonnes (sauf besoin réel de tout). Le `KanbanBoard` charge inutilement `extracted_data` (~25 KB/dossier) si on fait `select('*')`.
- **Polling sans condition d'arrêt** : tout `setInterval` ou React Query `refetchInterval` doit avoir une condition explicite qui le stoppe quand le travail est fini (cf. `ProcessingStep.jsx:74-82` pour le bon pattern).
- **Logique métier dans `useEffect`** : préférer un hook custom (`useAnalysis`, `useDocuments`) ou une mutation React Query.
- **Coercion de type ad-hoc** : utiliser les helpers `toNum/toDate/toChar1` (à factoriser dans `_shared/coercion.ts` côté serveur, `lib/format.js` côté client).
```

### Ajout 4 — Section "Edge functions : convention service_role"
```markdown
## Edge functions et service_role
Toute edge function utilisant `SUPABASE_SERVICE_ROLE_KEY` (= bypass RLS) DOIT :
1. Vérifier explicitement l'ownership de la ressource cible (session_id pour B2C, pro_token pour B2B).
2. Logger l'appel dans `pv_ai_logs` (utiliser `logging.ts:logAiCall()`).
3. Retourner 403 (pas 401, pas 500) en cas de tentative d'accès cross-tenant.
Les fonctions actuelles à refactorer pour respecter cette règle : `pv-run-extraction`, `pv-pro-credits`, `pv-create-payment-intent` (verify-checkout).
```
