# Audit RLS — Pack Vendeur (pre-etat-date.ai)

**Date** : 2026-04-26
**Project Supabase** : `odspcxgafcqxjzrarsqf` (partagé avec Majordhome)
**Schéma** : `pack_vendeur` (tables) + `public` (vues `pv_*`)
**Méthodologie** : Lecture seule via MCP Supabase. `pg_policies`, `pg_class`, `information_schema`, `storage.buckets`.
**Aucune** policy modifiée pendant cet audit.

---

## 🚨 Verdict en une phrase

**La sécurité de la couche données du projet est aujourd'hui équivalente à des tables sans RLS.** Toutes les policies anon des tables `pack_vendeur.*` sont écrites avec `USING (true)` et `WITH CHECK (true)` — c'est-à-dire **aucune restriction**. De plus, les vues `public.pv_*` exposées au client sont en mode `SECURITY DEFINER` (mode par défaut Postgres), ce qui **bypass complètement** les RLS de la table sous-jacente. Avec uniquement la clé anon publique (visible dans le bundle JS), un attaquant peut lire, modifier et supprimer **toutes** les données de **tous** les utilisateurs sans aucun token, sans payer, sans être pro. Le bucket Storage `pack-vendeur` est en plus déclaré `public=true`.

La protection actuelle ne tient que sur le fait qu'aucun attaquant n'a essayé. **Tout `access_token` qu'on implémente côté edge functions ne servira à rien tant que ces policies ne sont pas réécrites** : l'attaquant ignorera les edge functions et tapera directement la base.

### ⚠️ Le piège des vues `pv_*` (CRITIQUE)

Les 8 vues `public.pv_*` ont `reloptions = NULL`, ce qui veut dire qu'elles tournent dans le **mode par défaut Postgres**, équivalent à `security_invoker = false` (`SECURITY DEFINER`). En conséquence :

- Quand le client front fait `supabase.from('pv_dossiers').select('*')`, la requête s'exécute avec les **permissions du propriétaire de la vue** (typiquement `postgres`), **pas** celles de `anon`.
- Les RLS de `pack_vendeur.dossiers` ne s'appliquent **pas** via la vue.
- Seuls les `GRANT` sur la vue elle-même comptent.

**Vérification** des grants sur les vues : `anon` a `SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER` sur **toutes** les vues `pv_*`. Idem pour `authenticated` et `service_role`.

➡️ **Conséquence** : peu importe que les RLS sur `pack_vendeur.*` soient mal écrites — quand bien même elles seraient parfaites, elles seraient bypass par les vues. **Il faut corriger les deux** : (1) `WITH (security_invoker=true)` sur chaque vue, et (2) durcir les RLS sous-jacentes.

## 2. Détail policy par policy

Tous les rôles ciblés ci-dessous sont notés tels que retournés par `pg_policies`. Rappel : le rôle `public` PostgreSQL est un **pseudo-rôle qui inclut tous les autres** (anon, authenticated, service_role) — une policy sur `public` s'applique donc aussi à `anon`.

### 2.1 `pack_vendeur.dossiers` (vue `public.pv_dossiers`)

#### Policies existantes

| Nom | Cmd | Roles | USING (qual) | WITH CHECK |
|-----|-----|-------|--------------|------------|
| `service_full_dossiers` | ALL | service_role | true | true |
| `auth_full_dossiers` | ALL | authenticated | true | true |
| `anon_insert_dossiers` | INSERT | anon | — | **true** |
| `anon_select_dossiers` | SELECT | anon | **true** | — |
| `anon_update_dossiers` | UPDATE | anon | **true** | **true** |

#### Verdict d'exposition pour anon

| Opération | Anon peut ? | Restriction |
|-----------|-------------|-------------|
| SELECT | 🔴 OUI | **Aucune** — `qual=true`. Lit les 60 dossiers, tous champs (email vendeur, adresse, données financières, `share_token`, `stripe_payment_status`, `extracted_data`, etc.). |
| INSERT | 🔴 OUI | **Aucune** — `with_check=true`. Peut insérer un dossier avec n'importe quel `session_id`, `stripe_payment_status='paid'`, etc. |
| UPDATE | 🔴 OUI | **Aucune** — `qual=true, with_check=true`. **Peut faire `UPDATE pv_dossiers SET stripe_payment_status='paid' WHERE id='<n'importe quoi>'`**. |
| DELETE | ❌ Pas de policy DELETE pour anon | **MAIS** : la vue étant `security_definer`, le `GRANT DELETE` sur la vue peut suffire à bypasser. À tester en pratique — non confirmable via MCP en lecture seule. |

#### Données fuitées si exploité
- 60 dossiers × tous les champs : `email`, `seller_email`, `seller_name`, `property_address`, `property_lot_number`, données financières complètes (`charges_*`, `budget_previsionnel`, `impaye_vendeur`, `dette_*`), `extracted_data` JSONB (analyse IA brute), `validated_data`, `share_token` (= accès notaire), `stripe_payment_intent_id`, `client_email`, `pro_account_id`, `upload_token`, `questionnaire_data` (~80 champs).
- **Catégorie RGPD** : données personnelles + données contractuelles + données financières → notification CNIL obligatoire en cas de fuite avérée.

## 3. Audit du bucket Storage `pack-vendeur`

### Configuration du bucket

```
id              = pack-vendeur
public          = TRUE   ← ⚠️ PUBLIC
file_size_limit = NULL   ← ⚠️ pas de limite
allowed_mime_types = NULL ← ⚠️ pas de filtre MIME
```

➡️ **Le bucket est `public=true`**. Cela signifie que **n'importe qui** peut télécharger un fichier dont il connaît le chemin (`https://<project>.supabase.co/storage/v1/object/public/pack-vendeur/<path>`), **sans même utiliser l'API authentifiée**, **sans clé anon**. Les "signed URLs" générées par le code (cf. `documentService`) sont donc inutiles pour la confidentialité — le contenu est techniquement public.

### Policies sur `storage.objects` pour le bucket

| Nom | Cmd | Roles | qual / with_check |
|-----|-----|-------|-------------------|
| `pack_vendeur_select` | SELECT | **public** | `bucket_id = 'pack-vendeur'` |
| `pack_vendeur_insert` | INSERT | **public** | `bucket_id = 'pack-vendeur'` |
| `pack_vendeur_update` | UPDATE | **public** | `bucket_id = 'pack-vendeur'` |
| `pack_vendeur_delete` | DELETE | **public** | `bucket_id = 'pack-vendeur'` |

### Verdict d'exposition pour anon

| Opération | Anon peut ? | Restriction |
|-----------|-------------|-------------|
| Lister fichiers | 🔴 OUI | Aucune. Liste les 339 objets du bucket (préfixés par `<dossier_id>/`). |
| Télécharger | 🔴 OUI | Aucune. Et même sans API : URL publique directe, donc indexable par moteurs de recherche si un share leak. |
| Uploader | 🔴 OUI | **Sans limite de taille** (`file_size_limit=null`). Risque : disk-fill / abus de quota Supabase. |
| Écraser | 🔴 OUI | Aucune. Peut substituer le PDF d'un autre dossier (intégrité compromise). |
| Supprimer | 🔴 OUI | Aucune. Peut supprimer tous les uploads des autres utilisateurs. |

> Note : le projet contient aussi 12 autres buckets (`certificats`, `contracts`, `meeting-transcripts`, `premium-sources`, `project-recordings`, `interventions`, `technical-visits`, `product-images`, `product-documents`, `user-workspace`, `invoices`, `baikal-workspace`) qui appartiennent à Majordhome — leurs policies sont scopées avec `auth.uid()` et `org_id`, et **leurs policies semblent correctement écrites** (filtrage par dossier d'organisation, par utilisateur). C'est uniquement le bucket `pack-vendeur` qui est ouvert.

## 4. Verdict global et risques identifiés

Réponses explicites aux 5 questions posées :

### Q1. Anon peut-il lister tous les dossiers de tous les utilisateurs ?
**OUI, sans restriction.**
```sql
-- Exécutable depuis le navigateur via le client JS Supabase + anon key
SELECT * FROM pv_dossiers;
```
→ Renvoie les 60 dossiers actifs avec **toutes leurs PII** : email, adresse, surface, données financières, `share_token`, `stripe_payment_intent_id`, `extracted_data` complet, `validated_data`, `questionnaire_data`. **C'est une fuite massive RGPD-qualifiable.**

### Q2. Anon peut-il UPDATE un dossier (avec ou sans session_id) sans passer par une edge function ?
**OUI, sans même connaître le session_id.**
```sql
UPDATE pv_dossiers SET stripe_payment_status = 'paid' WHERE id = '<n'importe quel dossier_id>';
-- puis appel à pv-run-extraction → l'edge function vérifie stripe_payment_status='paid', accepte
```
**Conséquence** : un attaquant peut déclencher l'extraction Gemini gratuitement sur le dossier de n'importe qui, ou marquer son propre dossier comme payé sans payer. **Tout l'investissement dans un access_token côté edge functions sera annulé** tant que cette policy n'est pas resserrée — l'attaquant ignore l'edge function et tape directement la base.

### Q3. Anon peut-il INSERT des dossiers ou polluer pv_ai_logs ?
**OUI, sans restriction sur les deux.**
- `pv_dossiers` : `anon_insert_dossiers` accepte n'importe quel payload, y compris champs Stripe et `extracted_data` arbitraire.
- `pv_ai_logs` : `anon_insert_ai_logs` accepte tout. Un attaquant peut générer 100k entrées avec `total_tokens=999999` pour fausser les rapports de coût Gemini.
- Bonus : `pv_pro_credit_transactions` accepte aussi tout, donc création de fausses transactions de crédit B2B.

### Q4. Anon peut-il DELETE des données ?
**OUI sur `pv_documents` et tous les fichiers du bucket.**
- `pv_documents` : `anon_delete_documents` (qual=true) → suppression libre.
- Storage `pack-vendeur` : `pack_vendeur_delete` (public, qual sans restriction d'ownership) → suppression libre.
- `pv_dossiers` : pas de policy DELETE pour anon, mais la combinaison `security_definer view + GRANT DELETE on view` à anon **pourrait permettre le contournement** — non vérifiable par lecture seule, à tester. Si confirmé, c'est aussi exploitable.
- **Scénario sabotage** : un concurrent automatise un script qui supprime tous les documents en cours d'analyse toutes les 5 minutes → service entièrement bloqué pour les utilisateurs.

### Q5. Anon peut-il manipuler le bucket Storage ?
**OUI sur les 4 opérations**, et le bucket est en plus déclaré `public=true` (donc lecture sans même utiliser l'API). Quatre vecteurs distincts :
1. **Énumération + téléchargement** de tous les fichiers de tous les utilisateurs (PDFs sources + PDFs générés + ZIPs).
2. **Upload** sans limite de taille → abus de quota / disk-fill.
3. **Écrasement** des PDFs générés d'autres dossiers (corruption → faux documents délivrés au notaire).
4. **Suppression** de tous les uploads → sabotage.

## 6. Plan de remédiation proposé (ordre d'urgence)

### Étape 1 — Stop-bleed (24h)
Réécrire les RLS pour que **tout l'accès anon passe par les edge functions service_role uniquement**. Concrètement :
1. Pour les 8 tables `pack_vendeur.*` : **DROP** toutes les policies `anon_*` et celles ciblant `public`. Conserver uniquement les policies `service_role` (et `authenticated` si on veut préparer le futur, sinon DROP aussi).
2. Côté front : supprimer tous les usages directs de `supabase.from('pv_*')` pour les tables sensibles → tout passer par les edge functions. Inventaire à faire dans `src/services/dossier.service.js`, `document.service.js`, `pro.service.js`, `tracking.service.js`.

### Étape 2 — Vues security_invoker (48h)
Re-créer chaque vue `public.pv_*` avec `WITH (security_invoker = true)` afin que les RLS de la table sous-jacente s'appliquent au caller.
```sql
ALTER VIEW public.pv_dossiers SET (security_invoker = true);
-- (× 8 vues)
```

### Étape 3 — Storage (48h)
1. Passer le bucket `pack-vendeur` en `public=false`.
2. Définir `file_size_limit` (ex: 50 MB) et `allowed_mime_types` (`application/pdf`).
3. Réécrire les policies storage pour qu'elles vérifient l'ownership : par exemple `(storage.foldername(name))[1]` doit matcher un dossier dont le `session_id` est validé.
4. Tester que le code front (download via signed URL) continue à fonctionner.

### Étape 4 — Auth-by-token (1 semaine)
Implémenter le pattern `access_token` discuté précédemment :
- Génération d'un token signé côté edge function lors de la création du dossier.
- Stockage côté navigateur en httpOnly cookie ou localStorage (avec acceptation du tradeoff).
- Vérification dans chaque edge function avant toute mutation.
- Et seulement à ce stade : autoriser à nouveau certaines opérations anon ciblées via RLS, **avec USING strictement scopé** au token (ex: `current_setting('request.jwt.claim.token') = dossiers.access_token`).

### Étape 5 — Hardening complémentaire
- Gérer les migrations DB en code (Supabase CLI + repo) au lieu du dashboard, pour qu'un audit comme celui-ci puisse se faire en lecture de fichier.
- Activer `forcerowsecurity = true` sur les tables sensibles pour s'assurer que même `postgres` (propriétaire des vues) respecte les policies.
- Mettre en place un monitoring d'anomalie (alerte si > N requêtes anon sur `pv_dossiers` SELECT depuis une même IP).

## 7. Limitations de cet audit

- **Test exploratoire en pratique non réalisé** : je n'ai pas fait un vrai `SELECT` ou `UPDATE` depuis le client anon pour confirmer empiriquement que les requêtes passent. Toute la conclusion repose sur la lecture des policies + grants. Recommandation : reproduire les 4 ou 5 scénarios critiques (Q1–Q5) dans un script Node minimal avec uniquement `VITE_SUPABASE_ANON_KEY` pour confirmation. Probabilité que ça passe : >95% selon les policies lues.
- **Migrations** : non disponibles dans le repo (gérées via dashboard/MCP). On ne peut pas tracer historiquement *qui* a créé ces policies trop permissives.
- **Functions Postgres / triggers** : non audités. Une fonction `SECURITY DEFINER` mal écrite pourrait offrir un autre vecteur. Hors scope ici.
- **Advisors Supabase** : sortie trop volumineuse (220 KB) pour être analysée en place, j'ai filtré sur Pack Vendeur sans pouvoir en extraire un résumé exploitable. À relancer dans un sous-agent dédié si besoin de croiser avec d'autres signaux.

## TL;DR pour le commit message ou Slack

> Audit RLS Pack Vendeur : la posture sécurité actuelle est l'équivalent de tables sans RLS. Toutes les policies anon sont `qual=true`, et les vues `pv_*` sont en `security_definer` donc bypass des RLS. Le bucket Storage est `public=true` avec policies ouvertes. Un attaquant avec uniquement la clé anon (publique) peut SELECT/UPDATE/DELETE tous les dossiers, tous les pro_tokens, tous les fichiers. Implémenter un access_token côté edge functions ne servira à rien tant que les RLS et grants ne sont pas réécrits. **Stop-bleed prioritaire avant toute autre feature.**
